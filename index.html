A. The Template Method Pattern via Functional Callbacks

The Code Evidence (CsvHelper.h): Your readAll method uses a generic template T paired with a lambda parser function: std::function<T(std::stringstream&)> parser.

Design Principle Proof: This perfectly implements the Template Method behavior. The invariant steps (opening files, stream checks, line iteration loops) are fixed within CsvHelper. The variant steps (how a line becomes a Staff, Customer, or Vehicle object) are deferred and injected via custom lambdas. This cleanly achieves Open-Closed Principle (OCP) compliance—you can add infinitely many new models to your system without ever modifying the core file-parsing engine.

B. Interface-Driven Abstraction & Polymorphism

The Code Evidence (Staff.h): You designed a clean polymorphic hierarchy using an abstract base class (Staff) equipped with a pure virtual method virtual std::string getRole() const = 0; and a virtual default destructor.

Design Principle Proof: This satisfies the Liskov Substitution Principle (LSP) and Interface Segregation. By utilizing dynamic runtime binding through overrides (Manager, Salesman, Surveyor), high-level authentication services can interface entirely with the Staff abstraction without needing to couple themselves to concrete, low-level roles at compile time.

C. Strict Encapsulation & Serialization Layouts

The Code Evidence (Staff.h): Data members (id, name, username, password) are strictly hidden behind protected boundaries, exposed uniformly via constant reference getters (const std::string& getName() const) to prevent accidental mutability or data leakage.

Design Principle Proof: It respects the Single Responsibility Principle (SRP) by making objects completely autonomous; the serialize() routine manages string conversion logic inside the entity block itself, maintaining structural encapsulation.
